# Keynote PLAN — AI と一緒に JavaScript エンジンを作った話

## 発表概要

- タイトル: 「AI と一緒に JavaScript エンジンを作った話」
- 対象: AI を使って業務効率化を進めているインフラエンジニア
- 時間: 30〜40 分
- 形式: デモ中心 + 開発プロセスの紹介

## 構成

### パート 1: デモ (10 分)

Playground を使って「普段のコードの裏側」を見せる。

1. `console.log(1 + 2)` → Tokens / AST / Bytecode / IR タブを切り替え
   - 「1 行のコードのために 4 段階の変換が起きている」
2. fibonacci(25) を TW / VM / JIT で切り替え実行
   - 「同じコードなのに 1000 倍速い。なぜ？」
3. IR Optimizations プリセットで最適化の before/after
   - 「コンパイラが自動でコードを書き換えている」

### パート 2: どうやって作ったか (8 分)

5 ステップで段階的に構築した話:
1. 文字列を読む (Lexer)
2. 構造を理解する (Parser)
3. 実行する (Tree-Walking)
4. 速くする (Bytecode VM)
5. もっと速くする (JIT)

### パート 3: AI との協業の実際 (10 分)

**開発サイクル: RESEARCH → PLAN → TODO → Implement → LEARN**

- 各ステップの説明と実例
- AI が得意だったこと / 苦手だったこと
- 具体的な失敗例 (`fn.closure` vs `fn.closureEnv` 等)

**「正解がない領域」でどうするか:**

1. 正解がないことを認める
2. 作れるところから作る (単体テスト)
3. ハーネスを作る (ガイドライン、E2E)
4. 外部の目で早めに検証 (QA、PM)

### パート 4: 数字と振り返り (5 分)

- 結果の数字 (1000x、215x、753 テスト)
- まとめメッセージ

### Q&A (5-10 分)

## やること

- [ ] スライド作成
- [ ] デモのリハーサル (Playground が安定して動くか確認)
- [ ] fibonacci のプリセットが TW/VM/JIT 切り替えで速度差が体感できるか確認
- [ ] 発表原稿のドラフト

## やらないこと

- Promise / async/await の内部実装の説明 (聴衆にとって難しすぎる)
- SSA IR / Phi ノード等の詳細 (コンパイラ専門家向けの話)
- V8 との性能比較 (jsmini は教育用なので勝負しない)
- 黒田さんのフレームワークへの過度な当てはめ (後付けの部分は正直に)

## キーメッセージ

1. **AI があれば、専門外でも「作って理解する」ができる**
2. **小さく作って、テストで守って、ベンチで確認する**
3. **AI の出力は検証する。正解を段階的に囲い込む**
