# AI と一緒に JavaScript エンジンを作った話 — 発表資料リサーチ

## 対象聴衆

AI を使って業務効率化を進めているインフラエンジニアの方々。
JavaScript の専門家ではない。

## この発表で伝えたいこと

1. **AI (Claude) と一緒にここまでのものが作れる** — 1 人 + AI で JS エンジンを構築
2. **AI との協業の実際** — どう指示を出し、どこで躓き、どう修正したか
3. **ソフトウェアの「仕組みを知る」面白さ** — 普段ブラックボックスのエンジンの中身

---

## 参考: 黒田さんの「システム開発における Agentic AI の理解」との関係

黒田 樹 (Indeed Recruit Technologies) の資料の主張:

> **「Agentic AI を用いた開発は、モデルの賢さに期待するゲームではなく、
> 推論の幅をどう制御し、どう検証可能にするかのゲーム」**

> **「生成は確率的でもよいが、受け止め方は確率的であってはならない」**

設計原則:
1. 推論幅を狭める
2. 推論を分割する
3. 強い境界を作る
4. 契約・テスト・観測で検証可能にする
5. 可逆性を持たせる

### jsmini は後付けでこの原則に「結果的に」沿っていた部分がある

| 設計原則 | jsmini でやったこと | ただし... |
|---|---|---|
| 推論幅を狭める | TODO でスコープ限定 | 最初から意識していたわけではない。大きい変更が壊れたから小さくしただけ |
| 推論を分割する | 1 TODO = 1 コミット | 試行錯誤の結果そうなった |
| 強い境界を作る | CLAUDE.md (規約) | 規約は最低限で、厳密な境界設計ではない |
| 検証可能にする | 753 テスト + test262 | test262 は「たまたまあった」もの。自分で作ったわけではない |
| 可逆性 | Git + テスト全パス | これは自然にやっていた |

### 正直な振り返り

- 黒田さんのフレームワークを **最初から意識してやったわけではない**
- RESEARCH は「V8 どうやってるか調べて」と聞いただけのことも多い
- AI が間違えたとき、test262 より **自分で動かして「おかしい」と気づく** 方が先だった
- 上手くいった理由を **後から説明するフレームワーク** としては使える
- ただし「こう設計すべき」と「結果的にこうなった」は違う

### 発表で使うなら

黒田さんの資料を引用しつつ:
> 「この設計原則を最初から意識していたわけではない。
> ただ、振り返ると結果的にこの原則に沿っていた部分がある。
> test262 という決定論ハーネスがたまたまあったのが大きかった。
> 皆さんの業務では、テストスイートやガイドラインを意識的に作ることで、
> jsmini で偶然上手くいったことを再現可能にできるかもしれない。」

## スライド構成案 (30〜40 分想定)

### 1. 導入 (3 分)

- 自己紹介
- 「AI と一緒に JavaScript エンジンを作ってみた」
- 「普段なにげなく書いている `console.log(1 + 2)` の裏側で何が起きているか？」

### 2. デモ: Playground で見せる (10 分) ← メインパート

**デモ 1: 1 + 2 の裏側**

Playground に `console.log(1 + 2);` を入力。各タブを順に切り替えて見せる:

- **Tokens タブ**: `[console] [.] [log] [(] [1] [+] [2] [)] [;]` — 「文字列を単語に分割」
- **AST タブ**: ツリー構造 — 「単語の並びから構造を理解」
- **Bytecode タブ**: `LdaConst 1, LdaConst 2, Add, ...` — 「CPU が実行しやすい命令列に変換」
- **IR タブ**: SSA 形式 — 「さらに最適化しやすい中間表現に」

→ 「たった 1 行のコードのために、4 段階の変換が起きている」

**デモ 2: 速度の違いを体感**

Playground で fibonacci(25) を実行。モード切り替え:

- Tree-Walking: 450ms — 「素朴に実行」
- Bytecode VM: 90ms — 「命令列にすると 5 倍速い」
- Wasm JIT: 0.4ms — **「1000 倍速い!」**

→ 「なぜこんなに差が出るのか？」を次のパートで説明

**デモ 3: 最適化の before/after**

IR Optimizations プリセットを選択:
- IR タブで最適化前/後を見せる
- `x * 2` がループ外に移動 (LICM)
- `x * 4` が `x << 2` に変換 (Strength Reduction)

→ 「コンパイラが自動でコードを書き換えて速くしている」

### 3. どうやって作ったか — 5 ステップ (8 分)

#### ステップ 1: 文字列を読む (Lexer)

```
"var x = 1 + 2;"
    ↓
[var] [x] [=] [1] [+] [2] [;]
```

人間が文章を「単語に区切る」のと同じ。プログラムも単語 (トークン) に分ける。

#### ステップ 2: 構造を理解する (Parser)

```
[var] [x] [=] [1] [+] [2] [;]
    ↓
VariableDeclaration
  └── x = Add(1, 2)
```

単語の並びから「これは変数宣言で、右辺は足し算」という木構造 (AST) を作る。

#### ステップ 3: 実行する (Tree-Walking)

```
木を上から順に辿る:
  "変数宣言だな" → "右辺を計算" → "1 + 2 = 3" → "x に入れる"
```

シンプルだけど遅い。木を毎回辿り直すのが無駄。

#### ステップ 4: 速くする (Bytecode VM)

```
命令列に変換してから実行:
  LdaConst 1    // 1 を積む
  LdaConst 2    // 2 を積む
  Add            // 足す → 3
  StaLocal 0     // x に入れる
```

木を辿らず、命令を順に実行するだけ。2〜5 倍速くなる。**Chrome の V8 もこの方式。**

#### ステップ 5: もっと速くする (JIT)

```
「この関数、100 回も呼ばれてる。毎回数値だな」
    → 数値専用の機械語 (Wasm) にコンパイル!
    → 1000 倍速くなる
```

- 実行中に「どんなデータが来るか」を観察 (型フィードバック)
- 観察結果に基づいて専用のコードを生成
- もし予想が外れたら元に戻す (Deoptimization)

### 4. AI との協業の実際 (8 分)

#### 開発サイクル: RESEARCH → PLAN → TODO → Implement → LEARN

毎フェーズ、同じ 5 ステップを繰り返した:

```
① RESEARCH  「まず調べろ」
     ↓
② PLAN      「全体方針を決めろ」
     ↓
③ TODO      「具体的なタスクに分解しろ」
     ↓
④ Implement 「1 TODO ずつ実装 + テスト + コミット」
     ↓
⑤ LEARN     「何を学んだかまとめろ」
```

**① RESEARCH — まず調べる**

```
自分: 「Promise を実装したい」
AI:   V8 / JSC / SpiderMonkey の Promise 実装を調査
      → RESEARCH-Promise.md にまとめる
```

- 他のエンジンがどう実装しているか
- ECMAScript 仕様で何が定義されているか
- 落とし穴 (thenable, microtask 順序 等)
- **実装前に「地図」を作る。いきなりコードを書かない**

**② PLAN — 方針を決める**

```
RESEARCH を踏まえて:
- 何を実装するか / しないか
- どの順番でやるか
- 最終的なゴールは何か
```

例: Promise.all / race は「エンジン内部の対応不要、JS で書ける」と判断 → 後回し

**③ TODO — タスクに分解する**

```
TODO-Phase23.md:
  23-1: Promise オブジェクト (state/result/reactions)
  23-2: Microtask キュー (FIFO)
  23-3: Promise.resolve / reject + VM 対応
  23-4: async/await (Lexer + Parser + TW + VM)
  23-5: 統合テスト + Playground
```

- 各 TODO は **1 コミットで完了できるサイズ**
- チェックボックスで進捗を管理
- **大きすぎるタスクは分割。小さすぎるタスクは統合**

**④ Implement — 実装する**

```
1 TODO につき:
  AI がコード実装 → テスト書く → 全テストパス確認 → コミット
  ↓
  次の TODO へ
```

- **テストが通らないとコミットしない**
- ベンチマークで速度を確認
- 途中で「これ設計がおかしい」と気づいたら TODO を修正

**⑤ LEARN — 振り返る**

```
LEARN-Phase23.md:
- microtask queue は JS 配列の push/shift で実装。V8 は C++ deque
- .then() は settled でも microtask 経由 (Zalgo-free)
- JSPI の suspend は 15.7x 速いが microtask がボトルネック
```

- 想定と違ったこと、失敗したこと
- V8 との差がどこで出たか
- **次のフェーズに活きる知見を残す**

#### なぜこのサイクルが AI と相性がいいのか

**AI の最大の課題: アウトプットの振れ幅**

AI にいきなり「Promise を実装して」と言うと、方向性がブレる。
仕様の解釈が違う、設計方針が一貫しない、余計な機能を足す、等。

このサイクルは **振れ幅を小さくする仕組み**:

- **RESEARCH** → 「何を作るか」の認識を揃える。AI と人間が同じ地図を見る
- **PLAN** → 「何をやらないか」を先に決める。AI が暴走しない
- **TODO** → 1 タスクのスコープを限定。AI の出力が小さいから検証しやすい
- **Implement** → テストで正しさを保証。AI が間違えても検出できる
- **LEARN** → 次のフェーズで同じ失敗をしない

**要するに: 範囲を限定し、範囲から漏れても修正できるようにする**

#### JS エンジンがこのアプローチに向いていた理由

幸いなことに、JS エンジンには **test262** という 公式テストスイート (1 万件以上) がある。
さらに V8 / JSC / SpiderMonkey という **3 つの実在する実装** が参考にできる。

- AI が書いたコードが仕様通りか → **test262 で検証**
- AI の設計判断が妥当か → **V8 の実装と比較**
- AI が見落としたエッジケース → **test262 の意地悪テストが検出**

つまり **「正解」が存在する領域** だからこそ、AI のアウトプットを検証・修正できた。
正解がない領域 (新規ビジネスロジック等) では、このアプローチはもっと難しくなる。

**24 Phase × 5 ステップ = 120 のドキュメント** がリポジトリに残っている。

#### 「正解がない領域」ではどうするか

jsmini では test262 (1 万件のテスト) という「正解」があった。
しかし皆さんの業務では **正解やお手本がないことの方が多い**。

そういう場合は、**段階的に振れ幅を狭めていく**:

```
振れ幅: 大 ──────────────────────────────→ 小

1. 正解がない        (何も制約がない状態)
     ↓
2. 作れるところから作る (入出力の正解を部分的に定義)
     ↓
3. ハーネスを作る     (全体の枠で守る)
     ↓
4. 外部の目で検証     (開発の外側で早めにチェック)
```

**1. 正解やお手本がないことは多い**
- 新規機能、社内ツール、前例のないもの
- AI に「いい感じに作って」と言っても、振れ幅が大きすぎる

**2. 正解やお手本として作れそうなところは作ってみる**
- 単体テストを先に書く — 「この入力にはこの出力」を 10 件でいいから定義
- サンプルデータを用意する — AI が検証に使える「小さな正解」
- jsmini では: テスト 753 件を 1 TODO ごとに書いた

**3. それでも大きくぶれるときのためにハーネスを作る**
- コーディングガイドライン — AI が従うべきルール (CLAUDE.md)
- E2E テスト — 個々の単体テストでは見つからない全体の不整合を検出
- jsmini では: test262 がハーネス。CLAUDE.md に「enum は使わない」「テストはコロケーション」等

**4. 開発の外側で早めに検証できるようにしておく**
- QA に早めに繋げる — AI が作ったものを「使う人」に早く触ってもらう
- 仕様確認会 — ディレクターや PM に「これ合ってますか？」を定期的に
- jsmini では: Playground で自分が触って「おかしい」を見つけた

| 段階 | jsmini | 一般の業務 |
|---|---|---|
| 正解がない | Phase 初期 | 新規機能の立ち上げ |
| 部分的に作る | テスト 753 件 | 単体テストを先に書く |
| ハーネスを作る | CLAUDE.md + test262 | ガイドライン + E2E |
| 外部の目で検証 | Playground で自分が触る | QA + PM 確認会 |

**AI は「正解を知っている」わけではない。
正解を段階的に囲い込んで、AI の振れ幅を狭めていくのが人間の仕事。**

#### AI が得意だったこと

- **コード生成** — 「こういう関数が欲しい」→ すぐ書ける
- **テスト生成** — 正常系 + エッジケースを網羅的に
- **調査** — 「V8 はどうやってる？」→ ブログ/仕様書から要約
- **リファクタリング** — 「この関数を分割して」→ 安全に分割

#### AI が苦手だったこと / 人間がやったこと

- **方向性の判断** — 「これは今やるべきか？」「やめとくか？」
- **バグの根本原因** — AI は「それっぽい修正」をするが、根本が違うことがある
  - 例: `fn.closure` を `fn.closureEnv` と typo → AI は動くコードを書くが変数名が違う
- **ベンチマークの解釈** — 数字が出ても「なぜその数字か」は人間が考える
  - 例: JSPI が VM と同速 → 「microtask queue がボトルネック」と気づいたのは議論の中
- **ツッコミ** — 「それ本当に既知の制限？直せるんじゃない？」→ 直せた

#### 上手くいったコツ

1. **小さく作る** — 大きな変更を一気にやらない。1 コミットで 1 機能
2. **テストで守る** — 753 テストが常にパスしていることを確認
3. **調査してから実装** — 「まず調べて」→ RESEARCH.md → 理解してから書く
4. **ベンチで検証** — 「速くなった？」を数字で確認。思い込みを防ぐ
5. **AI を疑う** — 出力を鵜呑みにしない。「この設計で合ってる？」と確認

### 5. 数字で見る結果 (3 分)

| 何をしたか | 効果 |
|---|---|
| Tree-Walking (素朴な実行) | baseline |
| Bytecode VM | **2〜5 倍速** |
| Wasm JIT | **1000 倍速** |
| ループ途中から JIT 切り替え | 1 回しか呼ばない関数でも **215 倍速** |

- テスト: **753 個**、全パス
- 構築: **24 Phase**、1 人 + AI
- コード: TypeScript のみ (C++ なし)

### 6. まとめ (3 分)

- **AI があれば、専門外でも「作って理解する」ができる**
  - V8 のソースコード (C++ 30 万行) は読めなくても、同じ仕組みを TypeScript で作れた
- **「作る」と「読む」では理解の深さが違う**
  - JIT が速い理由、なぜ Deoptimization が必要か、等は作って初めてわかった
- **AI との協業はペアプロに近い**
  - 「考える人」と「書く人」の分担。ただし AI の出力は必ず検証する
