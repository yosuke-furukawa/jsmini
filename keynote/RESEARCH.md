## ストーリーライン案

### 導入: 普段使っている JS、中身どうなってるの？

```js
console.log(1 + 2);  // → 3
```

このたった 1 行を実行するために、エンジンの中ではこれだけのことが起きている:

```
"console.log(1 + 2);"
    ↓ 文字列を単語に分割 (Lexer)
[console] [.] [log] [(] [1] [+] [2] [)] [;]
    ↓ 文法構造を解析 (Parser)
CallExpression(console.log, Add(1, 2))
    ↓ 実行 (Interpreter)
→ 3
```

**これを自分で作ったら、中身がわかるのでは？** → 作ってみた (AI と一緒に)

### パート 1: AI と一緒に作る

#### どう始めたか

- 「JavaScript エンジンを TypeScript で作りたい」と Claude に伝えた
- 最初は Lexer (単語分割) から。`var x = 1;` を `[Var, x, =, 1, ;]` に分割するだけ
- 小さく作って、テストを書いて、動いたら次のステップへ

#### AI との分担

| 自分 | AI (Claude) |
|---|---|
| 何を作るか決める | コードを書く |
| テストを考える | テストも書く |
| バグを見つける | 原因を特定して修正 |
| 「もっとこうしたい」 | 設計を提案して実装 |
| 仕様を確認 | V8 の実装を調査 |

#### AI が躓いたところ

- **Promise の `.then` が動かない** → クロージャの変数参照 (`fn.closure` を `fn.closureEnv` と typo) で半日ハマった
- **Wasm のスタック順序がおかしい** → Const と Param の emit タイミングが Wasm のスタックマシンと合わなかった
- **`let` 変数がクロージャから見えない** → `var` と `let` のスコープの違いを compiler が正しく扱えていなかった

→ **AI は「それっぽいコード」は書けるが、デバッグは人間が方向づける必要がある**

### パート 2: JavaScript エンジンの中身 (平易に)

#### ステップ 1: 文字列を読む

```
"var x = 1 + 2;"
    ↓ Lexer (字句解析)
[var] [x] [=] [1] [+] [2] [;]
```

人間が文章を「単語に区切る」のと同じ。プログラムも単語 (トークン) に分けるところから始まる。

#### ステップ 2: 構造を理解する

```
[var] [x] [=] [1] [+] [2] [;]
    ↓ Parser (構文解析)
VariableDeclaration
  └── x = BinaryExpression(1 + 2)
```

単語の並びから「これは変数宣言で、右辺は足し算」という構造 (AST: 抽象構文木) を作る。

#### ステップ 3: 実行する (遅いやり方)

```
AST を上から順に辿る:
  "変数宣言だな" → "右辺を計算しよう" → "1 + 2 = 3" → "x に 3 を入れる"
```

これが Tree-Walking。シンプルだけど遅い。**V8 はこの方式を使っていない。**

#### ステップ 4: バイトコードにする (V8 と同じ)

```
AST → Bytecode:
  LdaConst 1      // 1 をスタックに積む
  LdaConst 2      // 2 をスタックに積む
  Add              // 足す → 3
  StaLocal 0       // x に入れる
```

AST を一度「命令列」に変換してから実行する。命令が単純なので高速。V8 の Ignition がこれ。

#### ステップ 5: さらに速くする (JIT)

```
「この関数、100回も呼ばれてるな。しかも毎回数値だな」
    → 数値専用の高速バージョンを Wasm でコンパイル!
    → 次から Wasm 版を実行 (1000倍速い)
```

- **型フィードバック**: 実行中に「この関数は数値ばかり受け取る」と記録
- **投機的コンパイル**: 数値専用にコンパイル。もし文字列が来たら元に戻す (Deoptimization)
- V8 の TurboFan / Sparkplug がこれ

#### ステップ 6: Promise と async/await

```js
// 同期的に書けるけど、中身は非同期
async function getData() {
  const data = await fetch("/api");
  return data;
}
```

中身: `await` で実行を一時停止 → Promise が完了したら再開。
jsmini では Generator の仕組みを使って実現。V8 も同じアプローチ。

### パート 3: 数字で見る結果

| 何をしたか | 効果 |
|---|---|
| Bytecode VM を作った | Tree-Walking の 2〜5 倍速に |
| Wasm JIT を作った | **1000 倍速に** |
| ループ途中から JIT に切り替え (OSR) | 1 回しか呼ばれない関数でも **215 倍速に** |
| メモリアクセスをループ外に移動 (LICM) | **79 倍速に** |
| Promise + async/await を実装 | TW/VM 両方で非同期が動く |
| JSPI (Wasm Stack Switching) を試した | suspend 自体は 15 倍速いが全体は同速 |

**テスト: 753 個、全パス。** test262 (公式テスト) の Promise テストも 49% 通過。

### パート 4: AI 協業で学んだこと

#### AI が得意なこと

- **定型的なコード生成** — ボイラープレート、テスト、似たパターンの繰り返し
- **既知の設計パターンの適用** — 「SSA IR を作りたい」→ V8 の設計を参考に実装
- **調査・要約** — 「V8 の Promise 実装はどうなってる？」→ 仕様と実装を要約

#### AI が苦手なこと

- **微妙なバグの原因特定** — 「この変数名は `closure` であって `closureEnv` ではない」
- **複数の変更の相互作用** — f64 昇格 + Strength Reduction + codegen の組み合わせで壊れた
- **「やめるべき」判断** — 大きすぎる修正でも突き進もうとする → 人間が止める

#### 上手くいった進め方

1. **小さいステップ** — 1 TODO = 1 コミット。大きな変更を一気にやらない
2. **テストファースト** — 「まずテストを書いて」と指示すると品質が上がる
3. **ベンチマークで確認** — 「速くなった？」を数字で確認。思い込みを防ぐ
4. **調査させてから実装** — 「まず V8 がどうやってるか調べて」→ RESEARCH.md → 実装
5. **ツッコミを入れる** — AI の出力を鵜呑みにしない。「それ本当？」が大事

### 締め: ブラックボックスを開ける楽しさ

- 普段使っている `console.log("hello")` の裏側で何が起きているか
- V8 のソースコード (C++ 30 万行) は読めなくても、同じ仕組みを TypeScript で作れる
- AI があれば、自分の専門外の領域でも「作って理解する」ができる時代
